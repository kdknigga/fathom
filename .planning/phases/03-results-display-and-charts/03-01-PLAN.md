1. `analyze_results(comparison: ComparisonResult) -> dict` -- the main function:
   - Extract costs using `result.paid_on_time.true_total_cost` for PromoResult, `result.true_total_cost` for OptionResult
   - Sort by cost ascending; winner = first, runner_up = second
   - Compute savings as Decimal difference
   - For winner_caveats: since ComparisonResult.caveats is a flat list without option names, AND `generate_caveats` is per-option, the cleanest approach is to store per-option caveats in the display data. Iterate `comparison.results.items()`, for each option call `generate_caveats` with the original FinancingOption -- BUT we don't have the FinancingOption in ComparisonResult. Instead, filter `comparison.caveats` by checking if the option name appears in the caveat message text. If no name match, assign to "general" caveats shown on the hero card.
   - Actually, looking at the caveat messages ("Deferred interest: pay at least...", "Total interest ($X) exceeds..."), they do NOT contain option names. So we need a different approach: accept `options: list[FinancingOption]` as a second parameter to `analyze_results`, then call `generate_caveats(option, result)` for each to get per-option caveats. This is the reliable approach.
   - Build `options_data`: list of dicts with `name`, `is_winner`, `is_promo`, `result` (OptionResult), and for promos also `not_paid_result` (OptionResult)
   - Build `breakdown_rows`: for each component (Total Payments, Total Interest, Rebates, Opportunity Cost, Tax Savings, Inflation Adjustment, True Total Cost), create a row with values per option. Skip rows where ALL values are Decimal(0). For PromoResult, include both paid_on_time and not_paid_on_time values.

Use simple heuristics based on option type of winner (cash -> opportunity cost framing, loan -> interest framing, promo -> conditional savings framing).
