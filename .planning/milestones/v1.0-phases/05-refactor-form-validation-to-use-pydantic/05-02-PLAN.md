  <name>Task 1: Rewrite forms.py with Pydantic validation models</name>
  <files>src/fathom/forms.py</files>
  <action>
Rewrite forms.py to replace the hand-written validation with Pydantic models. Keep these existing pieces:
- `_CHECKBOX_FIELDS`, `_OPTION_FIELDS`, `_OPTION_INDEX_RE` constants
- `_to_rate`, `_to_money`, `_to_int` helper functions (used by build_domain_objects)

**New Pydantic validation models (all in forms.py per user decision):**

1. `SettingsInput(BaseModel)` -- validates global settings:
   - `return_preset: str = "0.07"`
   - `return_rate_custom: str = ""`
   - `inflation_enabled: bool = False`
   - `inflation_rate: str = "3"`
   - `tax_enabled: bool = False`
   - `tax_rate: str = "22"`
   - `@model_validator(mode='after')` to validate return rate: if custom is non-empty, must be valid number 0-30; if empty, preset must be in {"0.04", "0.07", "0.10"}. On failure, raise ValueError that will map to `settings.return_rate` key.

2. `OptionInput(BaseModel)` -- validates a single option:
   - All fields as `str` (raw form values): `type: str`, `label: str = ""`, `apr: str = ""`, `term_months: str = ""`, `down_payment: str = ""`, `post_promo_apr: str = ""`, `deferred_interest: bool = False`, `cash_back_amount: str = ""`, `discounted_price: str = ""`, `custom_label: str = ""`
   - `purchase_price: str = ""` -- passed through from top level so cross-field validation (down_payment <= purchase_price, discounted_price < purchase_price) can happen inside OptionInput's model_validator, keeping error locations at the correct `options.N.field` path naturally.
   - `@model_validator(mode='after')` for type-dependent validation: replicate EXACT same logic as current `_validate_option` function. Same error messages, same range checks (APR 0-40, term 1-360, etc.). Use `_try_decimal` helper for parsing. Raise ValueError with the user-facing message.
   - For cross-field checks (down_payment > purchase_price, discounted_price >= purchase_price), validate inside the model_validator using the purchase_price field.

3. `FormInput(BaseModel)` -- top-level form:
   - `purchase_price: str`
   - `options: list[OptionInput]`
   - `settings: SettingsInput`
   - `@field_validator('purchase_price')` to validate: required, must be valid Decimal, must be > 0. Error messages: "Purchase price is required and must be a number." and "Purchase price must be greater than zero."
   - `@model_validator(mode='after')` to enforce option count (2-4). Error if < 2 or > 4.

**Key design decisions:**
- Form input models store raw validated strings (per user decision)
- All errors collected in one pass via Pydantic's validation pipeline
- Custom error messages via `raise ValueError('...')` maintaining current user-facing quality
- `_try_decimal` kept as helper for validators

**New helper function:**

4. `pydantic_errors_to_dict(exc: ValidationError) -> dict[str, str]`:
   - Convert `exc.errors()` loc tuples to dot-notation strings
   - Strip "Value error, " prefix from messages using `msg.removeprefix("Value error, ")`
   - Skip empty keys from root-level errors
   - This is the bridge between Pydantic and the existing template error display

**Updated `parse_form_data`:**
- Same extraction logic from ImmutableMultiDict (indices, option fields, settings)
- Instead of returning raw dict, construct and return `FormInput(purchase_price=..., options=[OptionInput(purchase_price=pp, ...) for ...], settings=SettingsInput(...))`
- If validation fails, Pydantic raises ValidationError automatically
- Pass purchase_price into each OptionInput for cross-field validation

**New `extract_form_data` function:**
- Extracted from the current `parse_form_data` logic
- Returns the raw dict (same as current parse_form_data return value)
- Used by add_option/remove_option routes which need form data restructuring without validation
- This is the research recommendation (Option 2) for handling non-validating routes

**Updated `build_domain_objects`:**
- Change signature: `build_domain_objects(form: FormInput) -> tuple[list[FinancingOption], GlobalSettings]`
- Access form.purchase_price, form.options, form.settings as Pydantic model attributes
- Keep `_to_rate`, `_to_money`, `_to_int` helpers for string-to-Decimal conversion
- Logic stays the same, just reading from FormInput instead of raw dict

**Delete:** `validate_form_data` function and `_validate_option`, `_validate_return_rate` helpers (replaced by Pydantic validators).
